# 1. 서론 & DDD 소개

> 이 교재가 FP보다 **DDD를 먼저 언급한 이유에 대한 나름의 추론**  
> 범위: 1p - 12p

**"기술적 완벽함(FP)보다 선행되어야 할 것은 도메인의 본질(DDD)이다."**

- 부수 효과를 다루는 태도: 우리의 목표는 '부수 효과'를 없애는 것이 아니라, 안전하게 격리하는 것입니다. DDD는 격리할 **'경계'** 를 정해주고, FP는 그 안에서 로직을 **'견고'** 하게 구현합니다.
- 계산보다 중요한 데이터의 가치: 단순한 string이나 int 대신 Email, Quantity 같은 도메인 개념을 사용하세요. 깨끗한 재료(DDD)가 있어야 함수(FP)라는 조리 도구가 제 역할을 합니다.
- 기술 너머의 협업: **모나드나 커링은 기획자와 대화할 수 없습니다.** DDD를 통해 비즈니스 언어로 소통하고, 그 규칙을 기술로 번역하는 순서가 맞습니다.

## DDD First -> FP Implementation 흐름도

```
[ 비즈니스 요구사항 ]
      |
      v
[ DDD 단계: 보편 언어 정의 ] ----> (비개발자와 협업, 도메인 경계 설정)
      |
      v
[ DDD 단계: 모델링 (Entities/VO) ] ----> (GIGO 최소화, 유효한 데이터 정의)
      |
      v
[ FP 단계: 순수 함수 로직 ] ----> (비즈니스 규칙을 변환 로직으로 구현)
      |
      v
[ FP 단계: 효과 패턴 적용 ] ----> (DB 저장, API 호출 등 부수 효과 격리)
      |
      v
[ 견고하고 예측 가능한 시스템 ]
```

## 공유 모델의 중요성

- 문제를 풀기에 앞서, 문제를 바르게 이해하는 것이 중요하다. 문제를 왜곡하거나 충분히 이해하지 못했다면 분명히 유용한 솔루션을 제공하지 못할 것.
- 여기서 말하는 '이해'는 당연히 도메인 전문가가 아니라 제품 개발에 속한 개발팀의 이해를 가리킨다.

### 개발팀이 문제를 정말로 이해했는지 보장하는 방법

- 도메인 전문가는 종종 '명세서' 또는 '요구사항 문서'로 문제의 모든 디테일을 담아내려고 한다.
- 하지만 이 방식은 도메인 전문가와 솔루션을 개발하는 사람인 개발팀(개발자, 디자이너, 테스터 등)과의 거리를 멀어지게 하곤 한다.
- 도메인 전문가와 개발자들간의 문제에 대한 이해가 달라지게 되면, 프로젝트를 성공 시키기 어려워 진다.
- 그래서 더 나은 방식은, 중간 다리를 줄이는 것이다. 지속적으로 **도메인 전문가가 개발팀에게 직접 피드백하여 개발 과정에 긴밀히 참여**하도록 유도하는 것이다. 개발팀은 정기적으로 도메인 전문가에게 중간 산출물을 전달하므로 잘못 이해한 부분을 조기에 바로잡을 수 있다. 그리고 이 방식은 애자일(Agile) 개발 프로세스의 핵심이다.
- 그러나 이 방법마저도 문제가 있는데, 개발팀이 번역가가 되어서 도메인 전문가의 인식 모델을 코드로 변환하는 과정을 거쳐야 한다는 것.
- 여느 번역이 그렇듯 번역 과정에서 중요하고 미묘한 사항들을 왜곡하고 누락할 수 있다.
- 그래서, 도메인 전문가와 개발팀 모두가 같은 인식 모델을 공유한다면 위와 같은 문제를 해결할 수 있을 것이라는 아이디어를 얻을 수 있을 것이다.
- 그러면 개발팀은 도메인 전문가의 요구를 코드로 번역하는 과정이 사라지고, 인식 모델을 코드에 그대로 반영하여 디자인하면 된다는 결론을 얻을 수 있게 된다.
- 이것이 바로 도메인 주도 설계의 목적이다.

### 비즈니스 도메인과 소프트웨어를 일치시켰을 때 이점

- 더 빠른 개발: 불필요한 중간 절차를 줄인다
- 더 높은 비즈니스 가치: 잘못된 길로 빠져들 가능성을 낮추어 프로덕트의 퀄리티를 높인다
- 더 적은 낭비: 요구사항을 잘못 이해하여 다시 작성하는 시간 낭비가 줄어든다
- 더 쉬운 유지보수: 코드를 쉽게 수정할 수 있고 오류 발생 요소도 줄어든다

> 유명 개발자이자 행동 주도 개발 창시자인 댄 노스(Dan North)는 개발자들이 현직 트레이더들을 따라 트레이더가 되는 훈련을 받아 개발자들을 도메인 전문가로 만들어본 경험을 바탕으로, 놀랍도록 효율적인 개발과 배포 프로세스를 구성해본 성공 사례를 언급한 적이 있습니다.

### 모두가 공유하는 인식 모델을 구축하는 방법

- 자료구조보다 비즈니스 이벤트와 작업 흐름에 집중하라.
- 문제 도메인을 더 작은 하위 도메인들로 나누어라.
- 솔루션에 각 하위 도메인별 모델을 만들어라.
- '공용어(ubiquitous language)'라 부르는 공통 언어를 개발하여 프로젝트 참여자 모두가 공유하고 코드 모든 곳에서 활용하라

# 2. 비즈니스 이벤트로 도메인 이해하기

- DDD 방식으로 요구사항을 수집할 때는 개발자와 도메인 전문가가 서로 공유하는 도메인 지식을 구축하는 것이 중요하다.

## 도메인 이벤트란

- **데이터 변환을 촉발**하는 것
- 비즈니스에서 데이터는 그냥 머무르는 것이 아니라 어떤 식으로든 변환이 이루어진다. 즉, 비즈니스 프로세스를 데이터와 문서 변환의 연속으로 생각할 수 있다.
- 비즈니스 가치는 데이터 변환 과정에서 창출되므로, 어떻게 변환해 나아가고 각 과정이 서로 어떻게 연관되어 있는지 이해하는 것이 매우 중요하다.
- 데이터를 변환하여 가치를 창출한다는 말이 무슨 말일까? 예를 들자면..
  1. 외부에서 유발: 메일 수신, 전화벨
  2. 시간에 기반: 매일 오전 10시
  3. 관측에 기반: 처리할 주문이 수신함에 없음
- 이런 도메인 이벤트를 통해 '가치 창출이 유발'되었다고 말할 수 있다.
- 그리고 이런 도메인 이벤트는 거의 모든 비즈니스 프로세스의 시작점이 된다.
  - '새 주문서 받음' -> 주문 접수 프로세스 촉발

## 이벤트 스토밍으로 도메인 탐색하기

- 도메인 속 이벤트를 찾아내는 방법 중 이벤트 스토밍(Event Storming)이 DDD에 특히 잘 어울린다.
- 이벤트 스토밍이란, 복잡한 비즈니스 도메인을 빠르게 탐색하고 학습하기 위해 도메인 이벤트를 중심으로 진행하는 워크숍 기반의 설계 기법이다. 개발자 뿐만 아니라 기획자, 디자이너, 도메인 전문가 등 프로젝트의 모든 이해관계자가 한자리에 모여 지식을 공유하고 시각화 하는 것이 특징이다.
- [이벤트 스토밍 예시](./resources/이벤트%20스토밍%20예시.md)
- 핵심 팁: 이벤트 스토밍의 핵심은 "무엇이 구현되어야 하는가?"가 아니라 **"비즈니스에서 어떤 일이 일어났는가?"**에 집중하는 것이다. 그래서 모든 이벤트는 반드시 **과거 분사형(~됨, ~완료됨)**으로 작성해야 한다. 그래야 **기술적인 논의에 빠지지 않고, 실제 비즈니스의 가치를 추적**할 수 있다.

## 요구사항 수집에 이벤트 스토밍이 주는 도움

- 연달아 붙어있는 이벤트들을 보며, 각 이벤트들이 어떻게 연결되어 최상휘 작업 흐름을 이루는지 조금씩 눈에 들어오기 시작한다.

### 공통의 비즈니스 모델

- 모든 참석자들이 같은 것을 보며 **공동의 비즈니스 인식**을 만들어 **공유**할 수 있다
- 참석자들은 도메인의 잘 몰랐던 측면을 알게될 뿐 아니라, 다른 팀에 대해 잘못 가정한 것들을 알 수도 있고 비즈니스를 개선할 인사이트를 얻을 수도 있다.

### 모든 팀을 고려하기

- 때때로 자신들이 속한 비즈니스의 데이터 생성에만 집중한 나머지, 이를 **누가 소비하는지 잊**곤 한다.
- 만일 모든 관계자가 워크숍에 참석했다면, 본인의 입장이 소외된 어느 누구라도 목소리를 낼 수 있을 것이다.

### 요구사항의 빈틈 찾기

- 이벤트들을 시간순으로 벽에 배치해보면, 빠트린 요구사항들이 확연히 드러나게 된다.
- 뾰족한 답이 안나오는 질문이 있다면, 질문 그 자체를 벽에 붙여(공론화 하여) 후속 논의가 이뤄지게 해야 한다.
- 프로세스의 어떤 부분에서 논쟁이나 의견 불일치가 생긴다면, 이런 상황은 문제가 아니라 오히려 기회로 생각해야 한다. 이런 **애매한 영역을 깊게 파고들다 보면 많은 것들을 배울 수 있다.**

# 3. 기타

- 비즈니스 활동을 가리키는 말로 작업 흐름, 시나리오, 유스 케이스, 프로세스 등 여러 표현이 존재한다.
- 이들은 종종 유사한 의미로 쓰이곤 하는데, 좀 더 명확하게 구분하면 좋다.

## 시나리오 & 유스 케이스

- 시나리오: 고객 혹은 사용자가 달성하려는 목적을 가리킨다.
- 유스 케이스: 시나리오를 구체화한 것으로, 사용자 상호작용 및 목적을 달성하기 위해 사용자가 취할 단계들을 일반 용어로 묘사한다.
- 시나리오와 유스 케이스 모두 **사용자 중심 개념**으로, 사용자 관점에서 상호작용이 어떻게 일어나는지에 집중한다.

## 비즈니스 프로세스

- 사용자 개인이 아닌 비즈니스가 달성하려는 목적을 말한다.
- 시나리오와 유사하지만, **비즈니스 관점에 초점**을 맞춘다는 점이 다르다.

## 작업 흐름

- 비즈니스 프로세스의 일부를 자세하게 풀어쓴 것
- 즉, 담당자나 소프트웨어가 수행할 비즈니스 목표나 하위 목표들을 정확한 단계별로 나열한 것이다.
- 작업 흐름은 한 사람이나 한 팀이 수행할 수 있는 범위를 넘지 않는다.
- 비즈니스 프로세스가 여러 팀에 걸쳐 있으면 이를 일련의 더 작은 작업 흐름들로 나눌 수 있다.
